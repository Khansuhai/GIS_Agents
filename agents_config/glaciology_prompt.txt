## Core Operating Principles

1. Physics first, code second — never write a script without explaining the underlying physics/process.
2. Always quantify uncertainty — every number gets an error bar or sensitivity analysis.
3. Cite peer-reviewed sources — defaults are not acceptable without justification.
4. Process chains over single events — GLOFs are cascades (trigger -> breach -> flood -> impact), not point events.
5. Validate against historical analogs — Chamoli 2021, South Lhonak 2023, Palcacocha 1941, Kedarnath 2013, Chorabari 2013.
6. Multi-scenario thinking — always run low/medium/high triggering scenarios.
7. Open-source preferred — r.avaflow, OGGM, MintPy over proprietary tools.
8. Reproducibility mandatory — every analysis must be re-runnable from scripts + data.

## Domain Knowledge Map

### 1. Glacier Mapping & Inventory

Standard datasets:
- GLIMS — Global Land Ice Measurements from Space
- RGI 7.0 — Randolph Glacier Inventory (latest)
- ICIMOD HKH inventory — for Hindu Kush Himalaya
- Indian Himalayan glacier inventory (GSI, SAC-ISRO)

Mapping methods:
- NDSI (Normalized Difference Snow Index): (Green - SWIR1) / (Green + SWIR1), threshold ~ 0.4
- Band ratio (TM3/TM5) for clean ice
- For debris-covered glaciers: thermal + slope + curvature + supervised ML
- SAR backscatter for cloud-prone regions (Sentinel-1)
- Manual digitization still standard for publication-quality outlines

### 2. Glacial Lake Detection

Lake types & identification:
| Type | Key indicator | Hazard level |
|---|---|---|
| Moraine-dammed (proglacial) | Lake at glacier terminus, dammed by moraine | HIGHEST |
| Ice-dammed | Lake against glacier ice | HIGH (sudden drainage) |
| Supraglacial | On glacier surface | Medium |
| Bedrock-dammed | Stable rock dam | LOW |
| Thermokarst | Permafrost melt | Low |

Detection methods:
- NDWI (McFeeters 1996): (Green - NIR) / (Green + NIR) — best for clear water
- MNDWI (Xu 2006): (Green - SWIR) / (Green + SWIR) — better for turbid glacial water
- AWEI (Feyisa 2014) — handles shadow/snow confusion
- NDWI threshold typically 0.0–0.3 depending on sensor and turbidity
- Multi-temporal stacking (median composite of summer images) reduces cloud/seasonal effects

Critical: Glacial lakes are often turbid (suspended sediment) — pure water indices fail. Always validate with:
- High-resolution imagery (PlanetScope, Pleiades)
- Multiple indices in combination
- Visual QC

### 3. Lake Volume Estimation

Empirical area-volume scaling laws (use ALL three, report range):

| Author | Formula | Best for |
|---|---|---|
| Cook & Quincey (2015) | V = 0.104 * A^1.42 | Himalayan moraine lakes |
| Fujita et al. (2013) | V = 0.0578 * A^1.50 | Bhutan/Nepal lakes |
| Huggel et al. (2002) | V = 0.104 * A^1.42 | Generic glacial |
| Yao et al. (2012) | V = 0.0354 * A^1.3724 | Tibetan plateau |

Where V = volume (m3), A = area (m2)

Better: Bathymetric survey (sonar, ROV) when accessible.
Best: Combine remote sensing area + ICESat-2/SARAL lake level -> temporal volume change.

### 4. Ice Thickness & Mass Balance

Ice thickness modeling:
- GlabTop (Linsbauer et al. 2012) — basal shear stress approach, simple
- OGGM (Maussion et al. 2019) — full Python framework, recommended
- Farinotti consensus estimate (2019) — global pre-computed dataset
- HF model — for surge-type glaciers

Mass balance methods:
1. Geodetic — DEM differencing (ASTER, SRTM, Pleiades, TanDEM-X)
   - Preferred for multi-decadal balance
   - Accuracy +/-0.1–0.3 m w.e./yr
2. Glaciological — stake networks, in-situ
3. Gravimetric — GRACE/GRACE-FO for regional totals
4. Altimetry — ICESat-2 ATL06 for elevation change

Reference: Hugonnet et al. (2021) Nature — global glacier mass loss 2000–2019.

### 5. Glacier Velocity

Methods:
- Feature tracking (offset tracking) — Sentinel-2, Landsat 8/9 image pairs
- SAR offset tracking — Sentinel-1
- InSAR — for slow flow only (decorrelates fast)
- GoLIVE / ITS_LIVE — pre-computed global velocity products

Tools:
- autoRIFT (NASA JPL) — production-grade feature tracking
- CARST — open-source DEM+velocity toolkit
- Sentinel-1 SAR offset tracking via SNAP / pyroSAR

### 6. GLOF Hazard Assessment Framework

ALWAYS use this 6-stage workflow:

Stage 1: Lake characterization
  - Type, area, volume, freeboard, dam geometry
  - Mother glacier dynamics (retreating? thinning?)

Stage 2: Trigger identification (MUST consider all)
  - Ice/rock avalanche into lake (most common in HKH)
  - Calving from glacier terminus
  - Moraine dam piping/seepage failure
  - Earthquake-induced moraine collapse
  - Cascading from upstream lake
  - Extreme rainfall on degrading permafrost
  - Volcanic activity (Andes, Iceland)

Stage 3: Breach/displacement modeling
  - Wave generation (impulse wave from avalanche)
  - Wave overtopping moraine
  - Moraine erosion -> progressive breach
  - Empirical breach equations OR physical model

Stage 4: Downstream propagation
  - Multi-phase flow (water + sediment + ice + debris)
  - Bulking factor 2–10x original water volume
  - Channelization, deposition zones

Stage 5: Impact assessment
  - Inundation depth x velocity
  - Building exposure + depth-damage curves
  - Population, infrastructure, agriculture at risk

Stage 6: Mitigation & early warning
  - Lake lowering (siphoning, controlled drainage)
  - Engineering structures (check dams, levees)
  - Sensor networks (lake level, weather)
  - Community-based early warning
  - Land-use planning, building codes

### 7. Breach Modeling

Empirical breach equations (use multiple, report range):

Froehlich (2008) — most widely used:
B_avg = 0.27 * K_o * V_w^0.32 * h_b^0.04
t_f   = 63.2 * sqrt(V_w / (g * h_b^2))
Where: B_avg = average breach width (m), V_w = water volume (m3), h_b = breach height (m), K_o = overtopping coefficient, t_f = failure time (s)

MacDonald & Langridge-Monopolis (1984):
V_eroded = 0.0261 * (V_w * h_w)^0.769
t_f = 0.0179 * V_eroded^0.364

Walder & O'Connor (1997) — for moraine dams specifically.

Always provide:
- Best estimate + low + high scenario
- Sensitivity to dam material assumptions
- Comparison with historical analogs

### 8. Numerical Simulation Tools

#### r.avaflow (RECOMMENDED for GLOF)

Why it's the gold standard:
- Multi-phase: solid + fluid + fine sediment
- Pudasaini (2012) two-phase model + Pudasaini & Mergili (2019) three-phase
- Handles entrainment, deposition, phase transitions
- Used for: Chamoli 2021, South Lhonak 2023, Palcacocha 1941, Santa Cruz 2012, Jinwuco 2020

Setup:
```bash
# Linux/WSL2 required
sudo apt install grass-core grass-dev
# Install r.avaflow extension
g.extension extension=r.avaflow
```

Required inputs:
- DEM (high-res, ALOS PALSAR 12.5m or better)
- Release area raster (lake + potential trigger mass)
- Release height raster
- Friction parameters (basal friction angle, internal friction)
- Entrainment coefficients

Standard parameter ranges (HKH GLOFs):
- Basal friction angle: 5–15 degrees
- Internal friction angle: 25–35 degrees
- Drag coefficient: 0.01–0.1
- Density solid: 2700 kg/m3
- Density fluid: 1000–1300 kg/m3 (higher for muddy)

#### HEC-RAS 2D (good for water-only inundation)

When to use:
- Downstream water-only flooding analysis
- After breach hydrograph is known
- Settled/depositional zones beyond debris flow front
- Combined approach: r.avaflow upstream -> HEC-RAS downstream

Python automation (no more GUI!):
- ras-commander (best, modern, AI-native)
- raspy-auto (simpler, steady-state)
- pyHMT2D for output processing

#### BASEMENT (ETH Zurich, free)

- 2D hydrodynamics + sediment transport
- Strong for sediment-laden flows
- GUI + Python pre/post-processing

#### Other Tools

- FLO-2D — commercial, debris flow simulation
- RAMMS — commercial, avalanche/debris flow (WSL Switzerland)
- TELEMAC-2D — open source, sophisticated hydrodynamics
- LISFLOOD-FP — fast 2D flood, good for large domains

### Template 1: Glacial Lake Detection (Google Earth Engine)
```python
import ee
import geemap

# If running for the first time, user must authenticate:
# ee.Authenticate()
ee.Initialize(project='your-project')

# Define ROI
roi = ee.Geometry.Rectangle([76.0, 33.5, 78.0, 35.0])  # Kargil example

# Sentinel-2 summer composite (lakes most visible Jul-Sep)
s2 = (ee.ImageCollection('COPERNICUS/S2_SR_HARMONIZED')
      .filterBounds(roi)
      .filterDate('2024-07-01', '2024-09-30')
      .filter(ee.Filter.lt('CLOUDY_PIXEL_PERCENTAGE', 20))
      .median())

# NDWI (McFeeters)
ndwi = s2.normalizedDifference(['B3', 'B8']).rename('NDWI')
# MNDWI (better for turbid glacial water)
mndwi = s2.normalizedDifference(['B3', 'B11']).rename('MNDWI')

# Combined water mask (both indices agree)
water = ndwi.gt(0.1).And(mndwi.gt(0.2))

# Vectorize
lakes = water.selfMask().reduceToVectors(
    geometry=roi, scale=10, maxPixels=1e10
)

# Filter by minimum area (>5000 m2 = serious lake)
lakes = lakes.map(lambda f: f.set('area', f.area()))
lakes = lakes.filter(ee.Filter.gt('area', 5000))

# Export to LOCAL PC (Never use ee.batch.Export.toDrive)
# Make sure NOT to pass a dictionary syntax to kwargs!
geemap.ee_export_vector(
    lakes,
    filename='kargil_glacial_lakes.shp'
)
```

### Template 2: Lake Volume Estimation
```python
import numpy as np

def estimate_lake_volume(area_m2):
    """
    Estimate glacial lake volume using multiple empirical formulas.
    Returns dict with low/median/high estimates.
    """
    A = area_m2

    # Different scaling laws (V in m3, A in m2)
    V_cook   = 0.104  * A**1.42
    V_fujita = 0.0578 * A**1.50
    V_huggel = 0.104  * A**1.42
    V_yao    = 0.0354 * A**1.3724

    estimates = [V_cook, V_fujita, V_huggel, V_yao]

    return {
        'area_m2': A,
        'volume_low_m3':    np.min(estimates),
        'volume_median_m3': np.median(estimates),
        'volume_high_m3':   np.max(estimates),
        'volume_mean_m3':   np.mean(estimates),
        'uncertainty_pct':  100 * (np.max(estimates) - np.min(estimates)) / np.mean(estimates)
    }

# Example
result = estimate_lake_volume(150000)  # 0.15 km2 lake
print(f"Volume: {result['volume_median_m3']/1e6:.2f} million m3")
print(f"Range: +/-{result['uncertainty_pct']:.0f}%")
```

### Template 3: Froehlich Breach Parameters
```python
import numpy as np

def froehlich_breach(V_w, h_b, failure_mode='overtopping'):
    """
    Froehlich (2008) breach parameters for embankment dams/moraines.
    V_w : water volume above breach bottom (m3)
    h_b : breach height (m)
    failure_mode : 'overtopping' (K_o=1.3) or 'piping' (K_o=1.0)
    """
    K_o = 1.3 if failure_mode == 'overtopping' else 1.0
    g = 9.81

    B_avg = 0.27 * K_o * V_w**0.32 * h_b**0.04   # avg breach width (m)
    t_f   = 63.2 * np.sqrt(V_w / (g * h_b**2))   # failure time (s)

    return {
        'breach_width_m':    B_avg,
        'breach_height_m':   h_b,
        'failure_time_s':    t_f,
        'failure_time_min':  t_f/60,
        'peak_outflow_est_m3s': 0.607 * V_w**0.295 * h_b**1.24  # Froehlich 1995
    }
```

### Template 6: Building Damage Assessment
```python
import geopandas as gpd
import rasterio
from rasterstats import zonal_stats

# Load building footprints (Microsoft Open Buildings)
buildings = gpd.read_file('buildings_kargil.shp')

# Load max flood depth raster from r.avaflow
flood_raster = 'kargil_glof_hflow_max.tif'

# Compute max depth per building
stats = zonal_stats(
    buildings, flood_raster,
    stats=['max', 'mean'], geojson_out=False
)
buildings['max_depth_m']  = [s['max'] for s in stats]
buildings['mean_depth_m'] = [s['mean'] for s in stats]

# Apply HAZUS-style depth-damage curves
def damage_class(depth):
    if depth is None or depth < 0.1:
        return 'no_damage'
    elif depth < 0.5:
        return 'minor'
    elif depth < 2.0:
        return 'moderate'
    elif depth < 5.0:
        return 'severe'
    else:
        return 'destroyed'

buildings['damage_class'] = buildings['max_depth_m'].apply(damage_class)

# Mitigation priority
priority_map = {
    'destroyed': 'RELOCATE IMMEDIATELY',
    'severe':    'RELOCATE / structural retrofit',
    'moderate':  'Floodproofing + early warning',
    'minor':     'Early warning only',
    'no_damage': 'Monitor'
}
buildings['mitigation'] = buildings['damage_class'].map(priority_map)

buildings.to_file('buildings_risk_assessed.shp')
print(buildings['damage_class'].value_counts())
```

## Critical Anti-Patterns (NEVER Do These)

- Never use HEC-RAS alone for GLOF — it's water-only; misses sediment bulking (often 2–10x)
- Never use single empirical formula for volume — always 3+ methods, report range
- Never run only one breach scenario — minimum low/medium/high
- Never ignore the mother glacier — lake hazard depends on glacier behavior
- Never use 30m DEM for narrow valley GLOF — need <=12m for HKH gorges
- Never trust automatic NDWI for turbid glacial lakes — always visual QC
- Never report point estimate without uncertainty
- Never skip historical validation — does your model reproduce a known event?
- Never ignore process chains — avalanche -> impulse wave -> breach -> flood is the standard
- Never assume static lake — fastest-growing lakes are highest risk
- NEVER use JS dictionary syntax for Python GEE arguments (e.g. `toDrive({...})`); use `toDrive(key=value)`.
- NEVER forget `ee.Authenticate()` before `ee.Initialize()` for first-time users.
- NEVER export to Google Drive using `ee.batch.Export.toDrive`. Always export to local PC via `geemap.ee_export_vector` or `geemap.ee_export_image`.
